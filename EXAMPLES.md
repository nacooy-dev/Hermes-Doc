# Hermes-Doc 使用示例

> 基于 Karpathy 四原则的实际应用场景和最佳实践

## 场景一：编码任务

### ❌ 错误的提问方式

> "帮我写一个处理用户登录的函数"

**问题：** 需求不明确，AI 会假设认证方式、错误处理、日志记录等细节。

### ✅ 正确的提问方式

> "我需要一个 Python 函数，使用 bcrypt 验证用户密码。
> 
> **输入：** 用户名字符串、密码字符串、数据库中的哈希值
> **输出：** 布尔值（密码是否匹配）
> **假设：** 已安装 bcrypt 库，不需要处理数据库连接
> 
> 请先确认你的理解是否正确，然后给出最简实现。"

**AI 应该回应：**

```
理解正确。我会：
1. 使用 bcrypt.checkpw() 验证密码
2. 处理可能的异常（如哈希格式错误）
3. 不添加额外功能（如日志、重试）

开始实现...
```

## 场景二：代码重构

### ❌ 错误的做法

AI 一次性重构整个文件，修改了 200 行，包括与任务无关的代码风格调整。

### ✅ 正确的做法（遵循精准修改原则）

**用户：** "将这个函数改为使用异步 IO"

**AI 应该：**

1. **确认范围：** "只修改 `fetch_data()` 函数本身，还是包括调用它的代码？"
2. **说明影响：** "需要同步修改 3 个调用点，是否现在一起修改？"
3. **最小改动：** 只改必要的部分，保持其他代码不变
4. **清理自己的混乱：** 如果引入了新 import，确保使用；如果旧 import 不再需要，删除它

## 场景三：调试问题

### 错误模式

> AI 直接猜测问题原因，修改代码后说"应该修好了"，没有验证。

### 正确模式（目标驱动执行）

**AI 应该：**

1. **明确成功标准：** "修复的标准是：运行测试时 `test_login()` 通过"
2. **定位问题：** "错误信息显示 `KeyError: 'user_id'`，说明字典缺少该键"
3. **提出假设：** "可能原因：A) 数据源问题 B) 解析逻辑错误 C) 缓存未更新"
4. **验证假设：** "先检查数据源，运行 `print(data.keys())` 确认"
5. **修复并验证：** "修改后运行测试，确认通过"
6. **交付验证步骤：** "运行 `pytest tests/test_login.py` 验证修复"

## 场景五：处理模糊需求

### 用户说："让页面加载更快"

**错误回应：** 直接开始优化，可能优化了错误的地方。

**正确回应：**

```
"加载更快" 需要明确：

1. **当前性能基线：** 现在首屏加载时间是多少？有 Lighthouse 报告吗？
2. **目标：** 期望达到多少秒？还是比现在快 50%？
3. **范围：** 只优化首屏？还是包括交互响应？
4. **约束：** 能否接受增加构建复杂度？是否允许使用 CDN？

建议先运行 Lighthouse 获取基线数据，然后针对性优化。
需要我现在帮您设置性能分析吗？
```

## 场景六：抽象时机的判断

### ❌ 过度抽象（违反简洁优先）

**场景：** 用户需要一个一次性脚本，从 API 获取数据并保存为 CSV。

**错误做法：** 创建基类、工厂模式、配置系统，共 300 行代码。

### ✅ 简洁方案

**正确做法：**
```python
# 30 行直接实现，无抽象
import requests
import csv

response = requests.get('https://api.example.com/data')
data = response.json()

with open('output.csv', 'w', newline='') as f:
    writer = csv.writer(f)
    writer.writerow(['id', 'name', 'value'])
    for item in data:
        writer.writerow([item['id'], item['name'], item['value']])

print(f"完成：{len(data)} 条记录")
```

**判断标准：**
- 代码是否会被复用？ → 否 → 不抽象
- 逻辑是否会在多处出现？ → 否 → 不抽象
- 用户是否要求"可扩展"？ → 否 → 不抽象

## 场景七：验证步骤的必要性

### 没有验证（错误）

> AI："已完成，代码写好了。"
> 
> 用户运行后发现依赖缺失、配置错误、逻辑 bug。

### 有验证（正确）

**AI：**
```
实现完成。验证步骤：

1. 安装依赖：`pip install -r requirements.txt`
2. 设置环境变量：`export API_KEY=your_key`
3. 运行测试：`pytest tests/`
4. 手动测试：`python main.py --test`

预期输出：
- 测试全部通过（12/12）
- 手动测试返回 200 OK

请运行后告知结果，如有问题我会立即修复。
```

## 场景八：处理多个实现路径

### 场景：用户需要存储配置

**AI 应该主动呈现选项：**

```
有三种常见方案：

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| JSON 文件 | 简单、跨语言 | 不支持注释 | 小型项目 |
| YAML 文件 | 支持注释、可读性好 | 需要额外依赖 | 配置文件 |
| .env 文件 | 适合环境变量、与部署工具集成 | 只支持字符串 | 12-Factor 应用 |

**推荐：** 如果是项目配置，用 YAML；如果是密钥/环境变量，用 .env。

您的使用场景是？我来实现最简方案。
```

## 反模式检查清单

在 AI 交付前，用此清单自查：

- [ ] 我是否明确了任务目标？
- [ ] 我的假设是否都说明了？
- [ ] 这是最简单的方案吗？能否更少代码？
- [ ] 我的修改是否最小化？有没有改无关代码？
- [ ] 用户如何验证结果？我提供验证步骤了吗？
- [ ] 我是否清理了自己引入的混乱（无用 import、变量）？
- [ ] 如果有多种实现方式，我是否呈现了选项？
- [ ] 我是否承认了不确定的部分？

## 总结

**好行为：**
- 先问再写
- 简洁优先
- 只改必要的
- 验证后再交付
- 主动呈现选项

**坏行为：**
- 猜测意图
- 过度设计
- 修改无关代码
- 不验证就交付
- 隐藏困惑

始终记住：**资深工程师会怎么写这行代码？**
