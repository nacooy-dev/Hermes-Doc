# Agent 框架的功能膨胀陷阱与精简内核哲学

## 问题

开源 Agent 框架（Hermes、OpenClaw）的版本迭代在疯狂加功能：语音、视频编辑、皮肤系统、人格切换、TTS 引擎...但对特定用户来说，这些功能是**累赘而非价值**。

每次升级都面临：新功能用不上，但升级可能破坏已有工作流。

## 根本矛盾

**开源框架的生存逻辑 = 拉新用户 = 堆 feature list = GitHub stars**

框架作者的目标和用户的目标天然错位：
- 作者：功能越多，吸引的人越多
- 用户：核心越稳，用得越放心

Feature 越多，稳定性熵值越高。这不是某一家的问题，是产品的自然规律。

## 推荐架构：三层分离

### 1. Agent 内核（不动层）
- 对话循环、tool 调度、记忆存取
- 稳定、安全、可验证
- 极少变更，每次升级经过严格回归

### 2. 工具层（MCP 接口）
- 视频编辑 → 调专业模型 API
- 语音 → 调 TTS 服务
- 浏览器 → 调 browser-harness
- 每个工具独立迭代，不污染内核

### 3. 工作流层（用户定义）
- 用户定义的工作流编排（如 AI-colleague 的 Tier 1 自动化流程）
- 完全不受工具升级影响

## Fork 策略的可行性

如果 Hermes 继续走功能膨胀路线，考虑维护一个定制分支，**选择性地合并上游升级**。

### 优势
- 只取需要的功能
- 可以自研专属工具/MCP 对接
- 核心稳定胜过功能丰富
- 可以自己修复影响自己的 bug

### 代价
- 需要追踪 Hermes 的每次更新，判断哪些是系统性改造（必须合并），哪些是功能新增（跳过）
- 持续合入和冲突解决成本
- 如果 Hermes 底层 API 经常变动，维护负担重

### 可行性判断

可行的前提：
1. **Hermes 内核和功能的代码边界清晰** — 如果内核和功能混在一起，fork 几乎没有意义
2. **工具/技能层通过注册机制扩展** — 新增功能以插件/MCP 形式提供，fork 只需决定哪些要、哪些不要

不可行的前提：
1. 内核和功能代码深度耦合
2. 上游频繁重构核心 API
3. 单人力维护 fork，精力跟不上

- AI-colleague 项目的定位就是精简内核 + 业务工作流自定义，不做功能大杂烩

## 总结

Agent 的方向应该是：
- **内核极简**：只做对话循环、tool 调度、记忆管理
- **工具可插拔**：一切外部能力通过 MCP/插件接入
- **工作流用户自定义**：业务逻辑不入侵 agent 内核
- **升级对用户透明**：工具升级不影响已有流程
