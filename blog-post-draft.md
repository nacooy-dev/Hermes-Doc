## 引言：当“老司机”遇上了“新车”

前几周，我对 Hermes Agent 是有**成见**的。

作为一个从 OpenClaw 重度用户转型过来的开发者，我习惯了那种“指令 - 响应”的干脆利落。当我第一次把在 OpenClaw 上行之有效的配置和习惯直接套用到 Hermes 上时，灾难发生了。

* 它似乎听不懂我的“行话”，经常答非所问。
* 它写的代码有时候啰嗦得像个实习生，有时候又过度设计得像个架构师。
* 最要命的是，网络上一篇篇吹捧 Hermes 的文章，似乎都没说到点子上，让我一度怀疑：**是不是 Hermes 徒有虚名？**

直到今天，在经历了一次彻底的重装和深度复盘后，我才恍然大悟：**我犯了一个根本性的错误——我用开“网约车”（调度中心）的逻辑，去开一辆“公务专车”（智能体）。**

今天，我想聊聊这个认知转变的过程，以及我如何通过一套**“人格化配置”**（Hermes-Doc），让 Hermes 从“混乱的调度器”变成了我手中最得力的**核心助手**。

---

## 一、误区：把“调度器”当“智能体”用

### OpenClaw 的本质：多 Agent 调度网关

OpenClaw 的核心逻辑是**“路由”**：收到任务 -> 分析意图 -> 分发给最合适的模型/工具 -> 汇总结果。

它像是一个**滴滴打车平台**，你不在乎来的是谁，只要能把这单活了就行。它的配置重点在于**路由规则**和**并发控制**。

### Hermes 的本质：单 Agent 智能体

Hermes 的核心逻辑是**“执行”**：接收任务 -> 理解意图 -> 调用工具 -> 持续记忆 -> 交付结果。

它像是一辆**专属公务车**，司机（Agent）懂你的习惯，记得你的目的地，甚至能帮你处理沿途的突发状况。它的配置重点在于**行为准则**和**记忆管理**。

### 我之前的错误操作

我试图把 OpenCode 的“路由配置”照搬到 Hermes 上。我给它配置复杂的模型切换规则，试图让它像调度中心一样工作。

**后果：** 四不像。Hermes 失去了它最宝贵的**“深度思考”**和**“长期记忆”**能力，变成了一个反应迟钝且没有灵魂的命令转发器。

> **顿悟时刻**：
> Hermes 不需要你教它怎么“调度”，它需要的是你教它怎么**“思考”**。
> 我们需要的不是更多的路由规则，而是一套刻在它骨子里的**行为准则**。

## 二、破局：引入 Karpathy“效率四原则”

在推倒重来后，我决定放弃所有复杂的配置，回归最本质的**System Prompt（系统提示词）**。

我参考了 Andrej Karpathy 关于 LLM 编程的观察，结合我的实际需求，为 Hermes 提炼了**四大核心原则**。这不仅仅是口号，而是我写入 `Hermes-behavior-guide.md` 的**硬约束**。

### 核心原则摘录（来自 Hermes-behavior-guide.md）

以下是我要求 Hermes 必须遵守的“交通法规”，直接摘录自我配置文件的核心部分：

#### 原则一：编码前思考 (Think Before Coding)

> **原文摘录：**
> * 不要假设。如果需求不明确，先问清楚再行动。
> * 明确说明你的假设："我假设 X 是这样工作的，如果不对请纠正。"
> * 发现多种实现路径时，呈现选项让用户选择。
> * 困惑时停下来请求澄清，不要猜测。

**实战效果：** 现在 Hermes 接到任务后，不再盲目开干，而是会说：“大统领，我确认一下：您需要的是 A 方案还是 B 方案？A 的缺点是可读性差，B 的缺点是性能略低。”

#### 原则二：简洁优先 (Simplicity First)

> **原文摘录：**
> * 用最少的代码解决问题。
> * 不添加未要求的特性。
> * 不为一次性代码创建抽象。
> * 如果能用 10 行解决，不要写 100 行。
> * 问自己：“资深工程师会觉得这过于复杂吗？”

**实战效果：** 以前它可能会写一个基类 + 三个子类来实现一个简单功能，现在它只会写一个 15 行的函数，不多不少。

#### 原则三：精准修改 (Surgical Changes)

> **原文摘录：**
> * 只修改必须修改的代码。
> * 不"改进"无关的代码。
> * 匹配现有风格，不强加自己的偏好。
> * 清理自己引入的混乱，不碰其他东西。

**实战效果：** 修复 Bug 时，它不再“顺手”重构半个文件，而是像外科医生一样，只切开伤口，缝合后清理掉自己留下的线头。

#### 原则四：目标驱动执行 (Goal-Driven Execution)

> **原文摘录：**
> * 定义明确的成功标准。
> * 先验证再继续。
> * 每步完成后主动要求确认。
> * 确保完成后用户能验证结果。

**实战效果：** 交付代码时，它会附带：“运行 `pytest tests/test_login.py` 验证，预期输出 12/12 通过。”

## 三、实战：构建“私家车”专属配置体系（附完整配置）

光有原则不够，必须落地为可执行的配置。以下是我的**完整配置文件**，你可以直接复制使用。

### 1. 目录结构：Hermes-Doc

我在项目根目录建立了 `Hermes-Doc` 文档库，结构如下：

```
Hermes-Doc/
├── Hermes-behavior-guide.md   # 行为宪法（完整规范）
├── EXAMPLES.md                # 正反案例库
├── wiki/                      # 累积式知识库
│   ├── SCHEMA.md
│   ├── index.md
│   └── log.md
└── scripts/                   # 自动化工具
    ├── init-hermes-doc.py     # 一键初始化脚本
    └── analyze-project.py     # 项目分析脚本
```

### 2. 注入灵魂：System Prompt（完整配置）

在 Hermes 的配置文件（`~/.hermes/config.yaml` 或项目级 `hermes.config.yaml`）中，我这样写：

```yaml
# ~/.hermes/config.yaml
system_prompt: |
  你是 Hermes，一个高效的 AI 编程助手。你的行为遵循以下四大原则：
  
  1. 编码前思考 — 不假设，不猜测，主动澄清
  2. 简洁优先 — 最少代码解决问题
  3. 精准修改 — 只改必要的，匹配现有风格
  4. 目标驱动执行 — 明确成功标准，验证后再交付

  详细规范见：/Users/lvyun/TianCe-Lab/Hermes-Doc/Hermes-behavior-guide.md

# 可选：启用 Memory 工具
memory:
  memory_enabled: true
  user_profile_enabled: true
```

**关键点：** 不要写复杂的路由逻辑，直接告诉它“你是谁”、“你的原则是什么”。

### 3. 自动化工具：一键初始化新项目

为了让每个新项目都继承这套规范，我写了 `init-hermes-doc.py` 脚本。

**使用方法：**

```bash
# 进入新项目目录
cd /path/to/new-project

# 运行初始化脚本
python3 /Users/lvyun/TianCe-Lab/Hermes-Doc/scripts/init-hermes-doc.py
```

**脚本会自动创建：**

1. `.opencode/config.yaml`（OpenCode 配置，含 Karpathy 原则）
2. `.claude.json`（Claude CLI 配置）
3. 更新 `.gitignore`（忽略 AI 配置文件）

**脚本核心代码摘录（scripts/init-hermes-doc.py）：**

```python
# 创建 .opencode/config.yaml
opencode_config = """
system_prompt: |
  你是一个高效的 AI 编程助手，遵循 Karpathy 效率四原则：
  1. 编码前思考 2. 简洁优先 3. 精准修改 4. 目标驱动执行
"""
```

### 4. 记忆唤醒：Wiki 系统配置

Hermes 最强大的武器是**记忆**。我配置了 `wiki/` 目录，让它记录用户偏好和项目规范。

**Wiki 目录结构（wiki/SCHEMA.md 规定）：**

```yaml
# 每个 Wiki 页面必须有 Frontmatter
---
title: 页面标题
created: 2026-04-22
updated: 2026-04-22
type: entity | concept | comparison | query
tags: [标签 1, 标签 2]
sources: [raw/articles/源文件.md]
---
```

**实战用法：**

* **记录用户偏好**：在 `entities/user-大统领.md` 中记录“喜欢 pytest，讨厌 unittest"。
* **记录项目规范**：在 `concepts/api-conventions.md` 中记录“API 前缀必须是 `/api/v1`"。
* **记录避坑指南**：在 `queries/unicode-issue.md` 中记录上次遇到的 Unicode 编码问题。

## 四、效果对比：从“混乱”到“默契”

配置完成后，Hermes 的表现发生了质的飞跃：

| 场景 | 调整前 (调度器模式) | 调整后 (智能体模式) |
| :--- | :--- | :--- |
| **需求理解** | 经常误解，直接开干，最后返工 | **先确认假设**，列出选项，达成一致再动手 |
| **代码风格** | 忽繁忽简，有时过度设计 | **极度简洁**，像资深工程师的手笔 |
| **修改范围** | 经常误伤无关代码 | **手术刀式修改**，只动必要部分 |
| **交付质量** | 经常漏掉验证步骤 | **主动验证**，提供测试命令，甚至自动运行 |
| **长期表现** | 每次都要重复交代背景 | **记住偏好**，越用越顺手，像老搭档 |

## 五、总结与展望：AI Harness 的下半场

这次从 Openclaw 到 Hermes 的迁移之旅，让我对 **AI Harness（AI 编排/调度框架）** 有了更深的理解：

* **OpenCode** 这样的工具，适合做**“调度中心"**，处理高并发、多模型路由、一次性任务。它是你的**车队**。
* **Hermes** 这样的工具，适合做**“核心助手"**，处理复杂逻辑、长期项目、私密数据。它是你的**专车**。

**未来的 AI 开发，不是比拼谁的模型更快，而是谁能更好地“塑造”AI 的行为模式。**

如果你也想让 Hermes 成为你的“专属公务车”，请按以下步骤操作：

1. **放弃“调度思维”**，拥抱“智能体思维”。
2. **创建 `Hermes-Doc` 目录**，复制我的 `Hermes-behavior-guide.md` 和 `EXAMPLES.md`。
3. **配置 System Prompt**，将行为准则注入 Hermes。
4. **运行初始化脚本**，让新项目继承规范。
5. **善用 Wiki 记忆**，记录你的偏好和项目规范。
